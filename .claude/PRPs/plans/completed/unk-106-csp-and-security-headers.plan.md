# Feature: Content-Security-Policy + bundled security headers (UNK-106)

## Summary

Add `Content-Security-Policy` and five other security response headers to the Reverie
backend in a single PR, using a hash-based CSP (`'sha256-...'`) to permit one known
inline FOUC script without `'unsafe-inline'` or per-request nonce templating. The
hash is generated by a ~40-line custom Vite plugin at `frontend/vite-plugins/csp-hash.ts`
that reads the FOUC script from a committed source file
(`frontend/src/fouc/fouc.js`), injects it into `index.html` at a comment marker, and
writes `dist/csp-hashes.json`. The backend reads the sidecar at startup, gains
ownership of the SPA route via `tower_http::services::{ServeDir, ServeFile}`, and
applies an `axum::middleware::from_fn_with_state` uniform-headers middleware plus
per-router CSP layers. Reverie does not host a violation-reporting endpoint; CSP
advertises `report-to` + `report-uri` if and only if
`REVERIE_CSP_REPORT_ENDPOINT` is set.

## User Story

As an operator running a self-hosted Reverie instance
I want the application to ship secure HTTP response headers by default
So that I get an A+ Mozilla Observatory / securityheaders.com grade without custom reverse-proxy configuration.

## Problem Statement

Reverie ships with zero security response headers. The backend returns plain JSON / XML
responses with no `Content-Security-Policy`, `Strict-Transport-Security`,
`X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, or
`X-Frame-Options`. External auditors (Mozilla Observatory, securityheaders.com,
manual pentest) will flag this as a P1 gap. The design-system plan
(`.claude/PRPs/plans/design-system.plan.md`) is parked pending CSP because it introduces
an inline FOUC script that will be blocked under any meaningful CSP.

## Solution Statement

Own the HTML response in the backend (via `ServeDir`/`ServeFile` for `/assets/*` +
SPA fallback), emit the full security-headers block as a single
`axum::middleware::from_fn_with_state` middleware applied outermost on the
composite router, and differentiate CSP by route class (HTML vs API) via per-router
`.layer()` for matched routes + manual CSP injection in a single composite-level
fallback (dictated by Axum 0.8's constraint that `.merge()` panics when multiple
routers define `.fallback()`).

## Deltas from brainstorm (`plans/2026-04-23-csp-design.md`)

The design doc ships with six factual corrections that this plan applies. Each is
unambiguous — the doc was a brainstorm, research closed the gaps.

1. **Per-router `.fallback()` + `.merge()` panics in Axum 0.8.** The design's "Axum
   router fallback note" hedges on this; authoritative docs at
   <https://docs.rs/axum/0.8.8/axum/routing/struct.Router.html> state "only one of
   the merged routers can have a fallback … will result in a panic." This plan
   adopts the design's own Plan B: a single composite-level fallback that
   path-prefix-dispatches 404-JSON vs SPA-`index.html`, plus an explicit fix for
   the CSP-on-fallback-response gap (the advisor's catch — see #2 below).
2. **Composite fallback must set CSP manually.** Per-router `.layer(api_csp)` /
   `.layer(html_csp)` only tag responses from *matched* routes. A 404 JSON from the
   composite fallback bypasses those layers and ships without API CSP; the SPA
   `index.html` via the fallback bypasses the HTML CSP layer for the same reason.
   The fallback handler inspects `uri.path()` and writes the right CSP header on
   its response before returning. Uniform headers still apply (outermost layer).
3. **Vite 8's `transformIndexHtml` context does not expose `ctx.command`.** Per
   <https://github.com/vitejs/vite/blob/v8.0.7/docs/guide/api-plugin.md>, the
   documented ctx shape is `{ path, filename, server?, bundle?, chunk? }`. The
   design's `ctx.command === 'build'` check fails silently (always falsy → sidecar
   never written → backend panics at startup). Correct detection: capture
   `ResolvedConfig` via `configResolved(config) { resolvedConfig = config }` and
   check `resolvedConfig.command === 'build'` inside the handler.
4. **`SecurityConfig` sub-struct on `Config`, not fields on `AppState`.** The
   design says `AppState` gains `csp_html_header`, `csp_api_header`, `hsts_value`.
   The existing convention (see `config.rs:158-161` sub-struct pattern —
   `EnrichmentConfig`, `CoverConfig`, `WritebackConfig`, `OpdsConfig`) is
   load-bearing: `AppState` holds runtime handles + `Config` by value, not
   precomputed strings. Correct placement: a new `SecurityConfig` sub-struct at
   `Config::security`.
5. **`regex` is not transitively present.** The design states `regex` is "already
   transitively present via dev deps; promoted to direct." Explorer confirmed it
   is absent from both `[dependencies]` and `[dev-dependencies]` in `Cargo.toml`.
   Plan adds `regex = "1"` as a new direct runtime dependency.
6. **`axum::middleware::from_fn_with_state`, not `from_fn`.** The uniform-headers
   middleware reads precomputed strings (`csp_html_header`, `csp_api_header`,
   `hsts_value`) from `State<AppState>`. Correct API per
   <https://docs.rs/axum/0.8.8/axum/middleware/fn.from_fn_with_state.html> is
   `axum::middleware::from_fn_with_state(state.clone(), security_headers)` applied
   via `.layer(...)`.

Additionally, two scope commitments the brainstorm left ambiguous:

- **Dev `server.proxy` config is explicitly deferred.** `frontend/vite.config.ts`
  has no `server` block today. The frontend has no `fetch` calls yet, so dev
  proxy is not yet functional-blocking. This PR adds only `server.headers` for
  dev-relaxed CSP. The first PR that introduces frontend→backend fetch traffic
  owns the proxy config.
- **FOUC placeholder script ships in this PR.** A minimal no-op at
  `frontend/src/fouc/fouc.js` is required for the plugin → sidecar → backend
  startup chain to work. Real FOUC content lands in the design-system plan's
  D3.13 task; the hash changes, the sidecar rewrites, no infrastructure churn.

// REPLACE: backend/src/config.rs:337-349
// New implementation — accepts only "true"/"false" (lowercase, exact match).
// The existing helper accepts "1"/"yes"/"no"/"0" + case-insensitive; pre-MVP
// there are no operators to migrate, so we tighten now. Design D8 absorbs UNK-110.
fn parse_bool(var: &str, default: bool) -> Result<bool, ConfigError> {
    match env::var(var) {
        Ok(v) => match v.as_str() {
            "true" => Ok(true),
            "false" => Ok(false),
            _ => Err(ConfigError::Invalid {
                var: var.into(),
                reason: format!("expected 'true' or 'false', got '{v}'"),
            }),
        },
        Err(_) => Ok(default),
    }
}
```

## NOT Building (Scope Limits)

Explicit exclusions — each with rationale:

- **Vite `server.proxy` for `/api`, `/auth`, `/opds`.** Dev mode requires only
  `server.headers` for dev-relaxed CSP to satisfy this PR. Frontend currently has
  zero `fetch()` calls. The first PR to introduce frontend→backend traffic owns
  the proxy config.
- **Real FOUC script content.** The plan ships a placeholder. Design-system plan's
  D3.13 task replaces the content; infrastructure (plugin, marker, sidecar, backend
  startup validation) is unchanged.
- **COOP/COEP headers.** Reverie does not use SharedArrayBuffer / WASM threads.
  Adds future friction without benefit.
- **CORP header.** Same-origin design.
- **SRI for external scripts.** No external CDN deps exist.
- **Reverie-hosted CSP violation endpoint.** Operator owns the endpoint;
  Reverie only advertises.
- **Nonce-based CSP infrastructure.** Hash-based covers the one known inline
  script; nonce machinery would cost templating engine + per-request middleware
  + deployment-topology churn for no added security.
- **Reverse-proxy config in `docker-compose.yml`.** Deployment samples live in
  `docs/deployment/reverse-proxy.md`; compose file stays minimal.
- **UNK-110 (bool-parser standardisation) as a separate ticket.** Folded into this
  PR per D8 of the design. UNK-110 is closed when this PR merges.

  use crate::state::AppState;

  ///
  /// Applied OUTERMOST on the composite router. Sets HSTS (conditional on
  /// behind_https), X-Content-Type-Options, Referrer-Policy, Permissions-Policy,
  /// X-Frame-Options, and Reporting-Endpoints (conditional).
  ///
  /// CSP headers are attached by per-router `.layer()` (matched routes) or
  /// by the composite fallback handler (unmatched paths). This middleware
  /// does NOT set CSP.
  pub async fn security_headers(
      State(state): State<AppState>,
      req: Request<axum::body::Body>,
      next: Next,
  ) -> Response {
      let mut resp = next.run(req).await;
      let headers = resp.headers_mut();

      // Uniform (unconditional)
      headers.insert(header::X_CONTENT_TYPE_OPTIONS, HeaderValue::from_static("nosniff"));
      headers.insert(header::REFERRER_POLICY, HeaderValue::from_static("no-referrer"));
      headers.insert(

          HeaderValue::from_static(PERMISSIONS_POLICY_VALUE),

      // Conditional: Reporting-Endpoints
      if let Some(re) = state.config.security.reporting_endpoints_header_value() {
          headers.insert(HeaderName::from_static("reporting-endpoints"), re);
      }
      resp
  }

  import { createHash } from 'node:crypto';
  import { readFileSync, writeFileSync, mkdirSync } from 'node:fs';
  import { dirname, resolve } from 'node:path';
  import type { Plugin, ResolvedConfig } from 'vite';

  const MARKER = '<!-- reverie:fouc-hash -->';
  const FOUC_SOURCE = 'src/fouc/fouc.js';
  const SIDECAR_FILENAME = 'csp-hashes.json';
  // Standard base64 alphabet (RFC 4648 §4, with padding). CSP rejects base64url.
  const STANDARD_BASE64 = /^[A-Za-z0-9+/]+={0,2}$/;

  export function cspHashPlugin(): Plugin {
    let resolvedConfig: ResolvedConfig | undefined;
    return {
      name: 'reverie-csp-hash',
      configResolved(config) {
        resolvedConfig = config;
      },
      transformIndexHtml: {
        order: 'post',
        handler(html, ctx) {
          if (!resolvedConfig) {
            throw new Error('reverie-csp-hash: configResolved not called');
          }
          const foucPath = resolve(resolvedConfig.root, FOUC_SOURCE);
          const fouc = readFileSync(foucPath, 'utf8');

          // Injection-safety guard: inline <script> is terminated by </script>
          // (case-insensitive). Content that includes this literal would escape.
          if (/<\/script>/i.test(fouc)) {
            throw new Error(
              `reverie-csp-hash: ${FOUC_SOURCE} contains '</script>' — inline script injection would break the HTML.`,
            );
          }

          // Marker presence + uniqueness
          const markerCount = (html.match(/<!-- reverie:fouc-hash -->/g) ?? []).length;
          if (markerCount !== 1) {
            throw new Error(
              `reverie-csp-hash: expected exactly one '${MARKER}' in index.html, found ${markerCount}`,
            );
          }

          const scriptTag = `<script>${fouc}</script>`;
          const injectedHtml = html.replace(MARKER, scriptTag);

          // Hash the script BODY (not the surrounding tag) — CSP L3 hashes
          // the text content of the <script> element.
          const digest = createHash('sha256').update(fouc).digest('base64');
          if (!STANDARD_BASE64.test(digest)) {
            throw new Error(
              `reverie-csp-hash: digest '${digest}' is not RFC 4648 §4 standard base64 (CSP L3 requires standard alphabet with padding)`,
            );
          }
          const sriValue = `sha256-${digest}`;

          if (resolvedConfig.command === 'build') {
            const outDir = resolvedConfig.build.outDir;
            const sidecarPath = resolve(resolvedConfig.root, outDir, SIDECAR_FILENAME);
            mkdirSync(dirname(sidecarPath), { recursive: true });
            writeFileSync(
              sidecarPath,
              JSON.stringify({ 'script-src-hashes': [sriValue] }, null, 2) + '\n',
              'utf8',
            );
          }
          return injectedHtml;
        },
      },
    };
  }
  ```

- **GOTCHA**: Vite 8's `transformIndexHtml` context has `{ path, filename, server?, bundle?, chunk? }` — NO `command` field. Build-vs-serve detection uses `resolvedConfig.command` captured in `configResolved`. The design doc's `ctx.command === 'build'` is wrong.
- **GOTCHA 2**: `createHash('sha256').digest('base64')` returns STANDARD base64 (RFC 4648 §4) in Node. `'base64url'` would return URL-safe base64 without padding — CSP browsers silently drop the unknown source. The regex guard is defense-in-depth.
- **GOTCHA 3**: The plugin runs for *every* `index.html` transform, including dev-server HMR responses. In dev, we don't write the sidecar (checked by `command === 'build'`), but we still inject the script — keeping dev and prod behaviour identical except for the sidecar file.
- **VALIDATE**: Task 22's tests validate this. For quick check: `cd frontend && npm run build` produces `dist/csp-hashes.json` with a valid `sha256-*` hash; `dist/index.html` contains `<script>(function () {})();</script>`.
